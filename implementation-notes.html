2026-05-30 · orchestration setup

decisionDrive the build with a recon-first, integrate-sequential workflow

The spec's build order (D → A2 → A3 → A1 → B2 → A4 → C1 → C2 → B3 → C3) is dependency-aware and most epics touch overlapping UI surfaces (nav, board, routes). The Workflow primitive has no merge-back for parallel mutating agents, so a naive parallel implementation would collide. Chosen structure:

  • Fan out the genuinely-independent work: read-only per-epic recon (now), and later the disjoint pure-logic+test files (each a brand-new src/lib/*.ts + its __tests__), which don't conflict.
  • Adversarially verify every recon blueprint against the actual code before trusting it (refute-not-rubber-stamp pass per CLAUDE/ENGINEERING principle 12).
  • Integrate sequentially in the main loop for the shared UI surfaces, in build order, running the four gates as I go.

Why over alternatives: full-parallel worktree-per-epic was rejected (no convergence path + heavy file overlap); fully-serial hand-implementation was rejected because the pure-logic layer is genuinely parallelizable and TDD wants it built first.
