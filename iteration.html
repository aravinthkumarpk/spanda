Legible & manual execution

From dogfooding the live board, one thing is clear: spanda can run a task with an agent, but it can’t yet be driven by hand. You can’t move a card yourself, you can’t easily see what a running agent is doing, and when the agent has a question it hides in a bell. This iteration inverts the board’s defaults — manual status becomes the primary gesture, and every task gets one clean surface to watch and steer its execution.

One source of truth throughout: status changes go through PATCH /api/beats into the beads DB; the new surfaces are views and controls over it, never a second store.

★ Agent Task Board — the model we converged on

What I'm actually trying to do: spend my time on one thing — writing and refining specs. Everything downstream of a good spec gets handled by agents. I don't babysit execution. I glance at a board, see what's happening, and step in only when there's a real decision.

Two human gates: plan review and execution review. Two agent phases: plan and execute. I stay out of the middle, and I never lose visibility.

The Do lifecycle — on the initiative, in plain words

Applies to Do tasks. Two ✋ columns are “needs you”. A glance tells you where a decision is waiting.

  1. 1Openthe initiative exists with a spec
  2. 2Planyou mark it; grill-me front-loads every question and proposes the minimum tasks
  3. 3 ✋PlanRevpage shows objective · problem · recommended solution (pros/cons) · breakdown · acceptance · assumptions — you decide
  4. 4ExecRdyapproved → tasks created, queued, nothing running
  5. 5Startyour one “go” releases the batch
  6. 6Execagents execute the tasks in parallel (dependency-aware); status page shows live progress + “what's done”
  7. 7 ✋ExecRevwhen all tasks finish, you review the produced output in one sitting — you decide
  8. 8Doneapproved

Locked rules — fallback: best-guess + document + flag at execution review, interrupt live only when genuinely stuck. Reject: Plan No → re-grill the spec; Execution No → redo that task (default) with a “mis-planned → back to Plan” escape; your reject note is the instruction.

The reframe — Foolery is the board; Claude Code & Codex are the brain

We were overcomplicating — trying to make Foolery run the agents (native grill, native goal-loop, native dispatch). That's the hardest, riskiest part, and it's redundant: Claude Code and Codex are already the executors. So Foolery's job shrinks to a context board — take context, update context, view context — and the agents stay in their native chat.

phase 1
Context board + skill pack. Foolery shows the initiative/task board with lifecycle columns and editable context. Agents (Claude Code/Codex) drive the lifecycle from their chat using the skill pack over the REST API: pick → plan → move → execute → status. The status page lives in Foolery — agents write progress + “what's done” back to the card via the API, and you watch status in Foolery, never by tailing the chat. Every decision we designed still holds — realized through the agents, not built natively. Ships fast, low risk.
phase 2
Execute natively from Foolery. Once the board+context loop is proven, bring execution in-house: native dispatch, the goal-loop, live streaming, the approval bridge surfaced on the page. The hard part, done last, on a proven foundation.

Methodology — Superpowers + goal-following are the engine

The agent discipline is not invented; it's Superpowers: Plan = brainstorming (one question at a time, front-loaded) + writing-plans (objective, breakdown, verification); Execute = subagent-driven development + RED-GREEN-REFACTOR TDD in a worktree + verification-before-completion.

And the spec is the goal: each task runs under a goal-following loop (Claude /goal Stop-hook; Codex goals=true) where the acceptance criteria are the completion condition. That's what makes “best-guess and keep going, interrupt only when stuck” automatic rather than hoped-for — the agent can't quit until acceptance holds.

You see minimal tasks; the agent works in small verified increments underneath. Two granularities, one board.

Skills & references — two packs, grounded in proven sources

The skill packs aren't invented from scratch — each Spanda skill borrows from a proven reference. Two agent packs: do (the full lifecycle) and decide (brainstorm → decide → communicate → close). Coordinate & Follow-up stay manual. All skills are cross-agent (Claude Code + Codex) and drive the board over the REST API. Each skill sits on top of the API — a thin procedure that carries no state of its own; the board (via the API) is the single source of truth, so you can add, swap, or reorder skills without touching the board, and two agents never fight over state. Each thin skill invokes the installed reference (gstack, Superpowers) as a sub-procedure where present, with an inline fallback (ADR-0002).

E2E testing & sign-off — the borrowed quality bar

spanda-signoff · gstack qa / review / ship / land-and-deploy / canary
borrowed
The Execution review gate is not a vibe-check — it's an evidence-based sign-off lifted from gstack's ship discipline.
  • gstack qa — systematically QA the app, fix bugs atomically, re-verify; produces before/after health scores (Quick / Standard / Exhaustive tiers).
  • gstack review — catch the bugs that pass CI but blow up in prod: SQL safety, LLM trust-boundary violations, conditional side-effects.
  • gstack ship + land-and-deploy — tests + coverage audit + PR, then merge, wait for CI, verify production health.
  • gstack canary — post-deploy watch for console errors / perf regressions against a pre-deploy baseline.
  • lavra /test-browser + Spanda's own quality-fe-agent (browser-scout / e2e orchestrator) for real browser e2e.
sign-off = every acceptance criterion verified with evidence before Done

Why this shape — the compound-engineering principle

Every reference converges on the same ratio: “80% of the work is planning and review, 20% is execution” (compound-engineering). That's exactly why our model front-loads grilling at Plan and puts two human gates around execution — and why the board turns those gates into a glance, not a chore.

Per ADR-0001 we pick one primary per phase, collapsing six references to three in active use: Superpowers (the Plan / Execute / Decide spine), gstack (Sign-off + the Plan-review office-hours assist), and lavra (the Learn memory loop) — with goal-following (Claude /goal, Codex goals) wrapping Execute to hold the agent to acceptance. mattpocock (to-issues) and compound-engineering (the 80/20 philosophy itself) drop to optional inspiration.

In sync with beads — no new store (ADR-0003)

The board is beads (.beads/issues.jsonl via /api/beats). Every part of the model maps onto fields beads already round-trips. Beads persists the full workflow state as a wf:state:<state> label and metadata is an open dict — so the whole lifecycle and the plan/status records fit with zero schema changes.
