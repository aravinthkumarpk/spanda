Fixing the dogfood findings

A live dogfood of the 2.1 build surfaced a cluster of breakages. Each was RCA'd against the running runtime (not guessed) and resolved one decision at a time. The good news from RCA: the data is intact and the architecture is right — the failures were a broken state-filter, a removed bd subcommand, and UI affordances that regressed when Today became the default landing. Design record: docs/adr/0005 (the bd-wrapper contract).

Decisions from the grill

Q1 status = guided next-states by default + a "set to any state" override. · Q2 every item gets a status page (initiative = full, others = light), keyed off real classification. · Q3 Today = clean embed (strip the daily's own chrome, scope its CSS, one app bar) + design-review + metrics-clickable. · Q4 bd sync → capability-detected import/export, best-effort. · Q5 Add button → global header + a Project picker. · Q6 clicking a project → scoped Board (edit = secondary). · Q7 state=all/absent = no filter.

Wave A — backend / data unblockers (TDD-first)
F1

state-filter returns nothing

done
report
[2]/[6] empty board · Codex found no beats
rca
bd-queries.ts:140 — applyWorkflowFilters falls through to beat.state !== filters.state, so state=all means "keep only state===\"all\"" → 0; open/in_progress never match the workflow state. Omitting state returns 228.
fix
treat state of all/empty as no filter; keep queued/in_action → loom phase mapping; raw state = exact. Also fix the dogfood prompt to drop state=all.
verify
unit: applyWorkflowFilters over all / queued / in_action / raw / requiresHumanAction · live: ?state=all&_repo=… = 228
F2

bd sync write errors

done
report
[3] "unknown command sync for bd" · [1] can't update tasks
rca
bd v1.0.4 removed sync (now import/export); we call bd sync in beats-sync-runner:26, bd-internal:487, bd-update:258. The reconcile also wasn't happening, so DB↔jsonl drifted (55 vs 228).
fix
capability-detect sync; on bd≥1.0 use import (auto-heal / import-only) and import+export (full reconcile); best-effort — never fail a write on a sync hiccup (ADR-0005)
verify
unit: command selection falls back to import when sync absent; a sync failure doesn't propagate to the write · live: a status update succeeds, then DB and jsonl agree

Assumptions taken filled in as built

F1 (state-filter): state=all & empty now skip the filter via a one-line guard in applyWorkflowFilters (filters.state !== "all"); queued/in_action keep their loom-phase mapping; a raw workflow state stays exact. The dogfood Codex prompt is fixed to drop state=all.
F2 (bd sync): one tested mapper bdSyncCommands is the single source for the command names. Runtime sites target the installed bd≥1.0 (hasSync=false): reconcile = import+export, the write auto-heal = import, the post-write flush = export — best-effort (a flush/sync failure logs a warning and never fails the user's write). A cached bdHasSync probe is built & tested for legacy-bd fallback but not wired into the tight paths (installed bd has no sync).
