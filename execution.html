Where spanda is headed

Spanda has the execution primitives. The next moves: simplify the board to a basic kanban, turn /today into a launchpad that creates real work, and grow a strategic view of the projects above it. One source of truth throughout — tasks in the beads system; every new layer is a view or rollup over it, never a second database.

Epic A — make the Board the daily driver

The execution altitude. Today the board has nine tabs; cut it to a basic kanban and make tasks actionable. Most of the logic shipped in Phase 2 — this is the UI layer over it.

Shared Add / Update task form

A2 · keystone
nowpure layer done
As an operator, I want one form to Add and Update tasks, so capture and edits feel identical and I never relearn a screen.
Acceptance
  • Add opens a blank form; Update opens the same form pre-filled with the task's current values (edit in place)
  • Editable fields: title, description, acceptance, priority, labels (incl. bucket), profile
  • Bucket-aware validation: do→acceptance required; coordinate/followup→person required; decide→none
  • Add → POST /api/beats; Update → PATCH /api/beats/:id
  • Nothing saves with a required field blank — errors surface inline (fail-loud)
  • Update edits fields only; state changes happen on the Board / via Complete, not in this form
reuses the shipped quick-capture validate/normalize pure layer · build once, reused by A3 + B2 + Board capture

Epic C — Projects (bird's-eye)

The strategy altitude. A project is the unit above the task: a named bet with the work nested under it and a health rollup. Built generically — any PM's portfolio, on the same beads.

Project as a first-class unit

C1
next
As a PM, I want to group tasks under a named project, so my work is legible as strategy, not a flat list.
Acceptance
  • A project has: name, one-line bet/intent, owner, member tasks (via a project:* label namespace)
  • A task belongs to one project; unassigned tasks land in "unsorted"
  • Projects are derived from existing labels, not a second source of truth
  • Creating/renaming a project never mutates a task's own lifecycle

Portfolio view: health at a glance

C2
next

Acceptance

  • A health signal per project (moving / stalled / blocked) derived from task activity
  • Click a project → the Board filtered to its tasks
  • Given a project with no movement in N days, When the portfolio loads, Then it reads "stalled"

Backfill the parent hierarchy

D1 · data foundation
now
As an operator, I want every task parented under a Project, so the hierarchy the Board and Projects read actually exists.
Acceptance
  • The live DB has 0 parent edges today — all 79 open tasks are flat, grouped only by project:* (agent-studio 36 / ai-transformation 28 / personal 14)
  • One idempotent pass creates a Project parent bead per project:* value and sets parent on every tagged task
  • DB tarball-backed before any write (rollback = untar + repoint)
  • project:* kept as a read alias; the parent edge is the structural truth (per the data model decision)
  • Initiatives (the middle layer) are created over time as needed — not backfilled
a Project = a parent bead; uses the native parent field, not a sidecar store

North star

"Hierarchical work units are a good start. Now we want a bird's-eye view of the managed products — a strategic approach to product evolution, not just knee-deep tactical execution."

Spanda grows from one altitude (the task) to three: a simplified execution board (A), a triage launchpad (B), and a strategic portfolio of projects (C). One source of truth — tasks in the beads system — with new layers as views and rollups over it. It sits inside the Evolve Aravinth program: free the operator from N-level work so they can do N+1-level work.
