A verification run of the latest build (iteration 2.2). No prior knowledge needed — follow the steps and tick them off. ← what changed (iteration 2.2)

Spanda is a board for your real work. You spend your time on specs; agents do the work between two checkpoints that are yours. Everything you see is your live task list (the “beads” store) — Spanda is a clean window onto it, never a second copy.

The one idea: a Do initiative runs Open → Plan → Plan review → Execution → Sign-off → Execution review → Done. The two ✋ shaded steps are the only places that need you — everywhere else an agent is working. Your job is to Approve or Reject at those two gates.
This run fixes the six things that broke last time. Each was a real finding from the previous dogfood; the steps below call out was broken vs now, so you can confirm the fix:
  • F1 The Board was empty → now shows your real tasks.
  • F2 Edits / status changes didn’t stick → now they persist through to the store.
  • F3 No Add button → Add now lives in the top bar on every screen.
  • F4 Project cards had no way into a scoped board → the title is now a link into the project’s board.
  • F5 Today was a broken, double-barred page → now a clean single-bar embed.
  • F6 Status page only showed for some items → every item now has one, with an Override control.

F1 The Board — your current stack (now populated)

2Open the Board tab. Four columns, left-to-right: To do · Doing · Review · Done. Each card shows the title and its real state tag.

Was broken: the Board came up empty even though you had ~80+ active tasks — a bad state filter was hiding everything.

Now expect: the Board shows your real active tasks (closed/done ones aren’t cluttering it). An initiative the agent is checking shows “Signing off” in Doing; anything waiting on you sits in Review. The action button reads Start (not “Run”).

3Move a card by hand (drag & drop). Drag a card one column forward.

Expect: it moves and saves immediately. Dropping onto Done asks you to confirm first. A move the workflow doesn’t allow (e.g. jumping straight to Done) snaps back — by design.

F3F2 Add & edit — and it sticks

4Add a task. Hit the Add button in the top bar (it’s there on every screen now, not just the Board). Under Task type you get exactly four: Do · Decide · Coordinate · Follow-up. Pick Do, give it a title and Acceptance criteria (a Do task won’t save without them), then create.

Was broken: there was no visible Add button on most screens.

Now expect: a new Do initiative lands at Open (To do). It’s tagged to a project (defaults to Unsorted if you don’t pick one). Only Do runs the full gated lifecycle; Decide / Coordinate / Follow-up you just move by hand.

5Edit its status — then refresh. Open the task → edit → the Status field offers the current state plus the valid next states. Change it, save, then reload the page.

Was broken: edits looked saved but didn’t make it through to the store (a stale sync command), so they’d vanish on refresh.

Now expect: a “Task updated” toast, the card moves columns, and the change survives the reload — it’s written through to the real store.

The two gates — where you decide

6Open the Review tab. It lists only the initiatives waiting on your decision — the Plan review and Execution review gates. (Live agent interruptions are separate — under More → Escalations.)

Expect: each row shows a ✋ Needs you pill and its state. Nothing waiting? The empty state says so — and right now nothing of yours is parked at a gate, so an empty Review tab is correct, not a bug.

7Open any initiative. Click it — its detail pane is its status page: live state, “what’s done”, any pending question, the breakdown, and your Approve / Reject controls when it’s at a gate.

Was broken: the status page only appeared for items carrying a special “initiative” label — which real tasks didn’t have — so most items showed nothing.

Now expect F6: every item shows its status page. Approve advances it past the gate; Reject opens a note box (your note is the instruction the agent redoes against — no note, can’t confirm). Below the guided buttons there’s an Override status dropdown — the escape hatch to set any state directly when the guided moves don’t fit.

F4 Projects — the bird’s-eye view

8Open the Projects tab. Projects show as cards with a health badge and a count. Click the project title.

Was broken: a project card had no obvious way into its own board — the only click opened an edit dialog.

Now expect: the title is a link into the Board scoped to that project — only its initiatives & tasks — with a “Scoped to project … · clear” chip. The little pencil is the (secondary) edit affordance; “view all N →” jumps into the same scoped board.

Parallel track — drive the board from Codex (the skill pack)

The other half of the model: agents work the board from their own chat using the spanda-* skill pack — thin, cross-agent procedures that read/write the board over its REST API. You can run today's work from Codex while you watch it land on the Board/Review tabs above.

Two facts that make it work. The skills live as plain markdown at .claude/skills/spanda-* (mirrored at .agents/skills/) — Codex just reads and follows them. And the live API needs an explicit repo: every /api/beats call must carry _repo=$REPO against http://127.0.0.1:3210 (a bare call returns 500). The prompt below sets both, and reads the repo path from the registry so it’s never hardcoded.

9Open Codex in the repo (cd /home/deploy/code/spanda && codex) and paste this prompt. It's click-to-select.

You're in the Spanda repo. A portable skill pack lives at .claude/skills/spanda-* (mirrored at .agents/skills/) — thin procedures that drive the task board over its REST API. Use it to work today's tasks, and stop at the human gates. Environment (the live app + its real data): export FOOLERY_URL=http://127.0.0.1:3210 export REPO=$(curl -s "$FOOLERY_URL/api/registry" | jq -r '.data[0].path') Every /api/beats call MUST include &_repo=$REPO (a bare call returns 500 repo_path_missing). List with, e.g.: curl -s "$FOOLERY_URL/api/beats?state=all&_repo=$REPO" | jq '.data[] | {id,title,state,profileId}' Do this, narrating each step: 1. Read .claude/skills/spanda-board/SKILL.md and .claude/skills/spanda-pick/SKILL.md. Show me the board: what's Open, what's executing, and anything resting at a human gate (requiresHumanAction=true). 2. Pick the next Do initiative to work. Respect the gates: never advance a card past plan_review or implementation_review — those two decisions are mine. 3. Read .claude/skills/spanda-plan/SKILL.md and run it on that initiative: grill the spec (one question at a time), write the plan to metadata.plan, create the minimum child tasks, then PATCH the initiative to plan_review. Stop and hand it back to me for Plan review. 4. Wait for my approval, then read spanda-execute + spanda-signoff and continue (worktree + TDD; PATCH state + metadata.status as you go; move to sign_off then ready_for_implementation_review). Rules: the board (via the API) is the only source of truth — keep no state of your own. If you hit a question only I can answer, write it to metadata.question and keep going on independent work. Never set state=shipped — that's my Execution-review approval.

Expect: Codex lists today's initiatives, picks one, asks its planning questions, writes a plan, and parks it at Plan review — which then shows up in the Review tab of the web app (step 6). Approve/Reject there or tell Codex; the two of you share one board.
