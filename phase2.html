brainstorm 1workflow profiles (item #1)

The fork already supports custom BuiltinProfileConfig with customStates. Question: which set of profiles is the right one?

option aMinimum split — 3 profiles (code, coord, followup)rejected

Covers ~95% of corpus. Simplest mental model.

But: doesn't differentiate "decision pending" beads from coord beads. Both go through scheduled → done, but a decision has a different texture (you wait for someone to decide, then act).

option bFine-grained — 8 to 10 profilesrejected — overkill

code-review, doc-review, meeting, chase, decision, read, write, purchase, personal…

Too many profiles to mentally track. Settings UI would have 8+ profile dropdowns. Most categories collapse into 3-4 in practice.

option cDomain-driven — per MISSION.md goals (G0/G1/G2/G3)rejected — wrong axis

Each mission goal gets its own workflow.

But: workflows describe how work flows, not what it's about. G1 work can be code (autopilot fits) or coord (autopilot doesn't fit).

option dModerate split — 4 profiles (code / coord / followup / decision)recommended

Each profile has 2-4 columns max → board stays scannable. decision vs coord vs followup are textures of human work that need different next-action verbs (decide vs attend vs ping). The default code profile keeps the existing IC workflow unchanged so nothing breaks for agent-dispatchable beads.

ProfileStatesWhen to useOwner
code ready_for_impl → impl → ready_for_review → review → shipped IC work an agent can do agent
coord scheduled → done / cancelled Meetings, calls, alignment human
followup waiting → nudged → escalated → done / closed Chasing someone external human (+ nudge agent later)
decision deciding → decided → executed / dropped Choice YOU need to make human

Routing. Labels work:code / work:coord / work:followup / work:decision. No label = code (backwards-compat with existing 50 beads). Two work: labels = fail-loudly per CLAUDE.md.

Action verbs per profile (drives item #8 row-action work):

ProfileNon-terminal actionTerminal action
codeRun agent (was "Take!")Ship
coordMark scheduled / RescheduleMark done / Cancel
followupMark nudged / EscalateMark done / Close
decisionMark decidingRecord decision / Drop

brainstorm 2vocabulary (item #6)

DESIGN.md says the musical metaphor is "intentional and distinctive." User has decided in favor of accessibility for any knowledge worker. Remaining question is HOW.

option aFull rename across the codebaserejected — fork-tax trap

Touch every internal symbol: Beat → Task, useBeats → useTasks, /api/beats → /api/tasks.

~500-1000 LOC of churn across hundreds of files. Every upstream Foolery rebase becomes a mess. Breaks the API contract.

option bUI-only label-translation layerrecommended

Single source of truth at src/lib/ui-vocab.ts. Two vocabularies registered: plain (default) and verbose (musical originals, opt-in).

Components consume via useVocab(). All 17 user-facing strings swap from literal "Take!" to vocab["Take!"]. Internal types, API routes, file names stay on the musical names → zero upstream Foolery diff, zero API churn.

Plain mode mapping: Beat→Task, Take!→Run, Scene!→Plan it, Setlist→Plan board, ReTakes→Reviews, Escalations→Blockers, Capsule→Context.

Reversible (toggle to verbose in 1 click). Opens door to multi-language later (any vocab is a swap-in dictionary).

option cGlossary tooltips, no renamerejected — fails the bar

Keep "Take!" / "Scene!" labels but add a ? tooltip explaining each.

Doesn't actually fix the problem. New users still have to learn jargon BEFORE they can act.

option dSettings toggle — plain is opt-inrejected — defaults matter

Same as B but user has to flip a switch to get plain mode.

Discovery problem — new users won't find the switch.
