Firming the assumptions

Iteration 02 shipped with eleven decisions taken outside the written plan, recorded as “⊕ assumption” notes on that page. This iteration walked each one down to a deliberate, shared decision. The trunk: a “Do” initiative was silently mapped to an autonomous, gateless profile — the opposite of the two-gate model the whole product rests on. Fixing that cascaded through the board, the gates, the status page, and the glossary.

Design record: docs/adr/0004 supersedes the lifecycle/profile, status-surface, and altitude portions of ADR-0003. Status badges below flip to done as each decision is implemented & gate-green.

The do-profile trunk two human gates, restored

The contradiction: the form mapped task type Do → profileId: "do", but the do profile was a leftover rename of autopilot_no_planning — planning skipped, all-agent owners, zero human gates. We fixed the profile, not the mapping.

D1

Redefine do to be human-gated

done

The do profile becomes the canonical, gated “Do” lifecycle — not the autonomous leftover. (The autonomous no-planning flow stays available as autopilot_no_planning.)

replaces ⊕ A4 / B2: “Do = profileId do (autonomous)” vs ADR-0003 “Do = semiauto” workflows.ts: do → planningMode required, owners SEMIAUTO_OWNERS
D2

Exactly two human gates via SEMIAUTO_OWNERS

done

plan_review and implementation_review are human-owned; the agent owns planning, implementation, sign-off, shipment. The human reviews; the agent works.

refines ⊕ Q2 interview do.owners = SEMIAUTO_OWNERS; implementation_review stays human (before ship)
D3

An explicit agent sign_off state

done

The agent’s “check first” (qa / review / canary) becomes a real lifecycle state before the human gate — not a note in free text. Added via a signOffMode flag scoped to do (the shared SDLC graph is untouched). Renders in Doing as “Signing off”.

implementation (agent) → sign_off (agent — spanda-signoff) → implementation_review (human, final pass) → shipment → shipped
refines ⊕ B4 “sign-off as metadata.status” → explicit state workflows.ts: signOffMode flag + sign_off state/transitions; board-columns → Doing
D4

Keep semiauto as the compat id

done

do is the one spanda task-type profile; semiauto stays untouched as the knots/upstream coarse id. They’re no longer twins (do has sign_off).

resolves the do-vs-semiauto duplication flagged in Q1 no change to semiauto; SPANDA_TASK_TYPES Do→do now correct as-is

Gates & surfaces where the human acts

D5

“Review” tab = the human-gate queue

done

Review lists initiatives waiting on your decision (requiresHumanAction=true) — Plan review + Execution review. The legacy escalations (finalcut) view, a different execution-time concern, moves into More.

replaces ⊕ A5: “Review → finalcut/escalations” (a conflation) new "review" view (filter requiresHumanAction); finalcut → More
D6

Gate decisions = explicit Approve / Reject (not drag)

done

Approve / Reject controls on the Review queue + status page. Reject captures a note (“your reject note is the instruction”); Execution-review reject offers redo-task (default) or mis-planned→back-to-Plan. Drag stays single-step only (mark-for-plan, Start, advance, terminal-confirm).

refines ⊕ A2: drag never addressed the gate decision (a branch + a reason) Approve=forward transition, Reject=rollback (isRollbackTransition) + notes
D7

Stamp altitude at create

done

Stamp altitude:initiative on a top-level beat created via the add-task form / promote. The label only disambiguates the childless case — once a beat has children, structure decides (so a stamped initiative that grows children is still seen as a container, and the projects view is preserved). Tasks aren’t stamped; altitude:project is reserved for a future explicit grouping action (move-to-project is a cross-repo move, not a grouping). classifyBeatRole is the fallback.

refines ⊕ B1: the fix only fires if creation stamps the label — and label must not freeze a grown beat create-beat-dialog stamps altitude:initiative when no parent; classify: label disambiguates childless only
D8

Status page = the initiative’s detail pane

done

BeatDetail branches on altitude: an initiative renders StatusPage (state, what’s-done, sign-off evidence, pending question, task breakdown, Approve/Reject); a task renders the existing detail. Reuses /beats?beat=<id> — no new route. metadata.question = async note; escalations = sync.

replaces ⊕ B4: a component with no home → the initiative’s real surface mount StatusPage in BeatDetail behind an altitude check + gate controls
